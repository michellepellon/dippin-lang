DOT wasn't built for this

Graphviz DOT is great for graph visualization. But authoring AI pipelines with multi-line prompts, typed nodes, and conditional edges? It falls apart.

Graphviz DOT 
digraph Pipeline {
  Planner [kind="agent"
    llm_model="claude-opus-4-6"
    prompt="You are a senior architect.\nAnalyze the request and\nproduce an implementation plan."];
  Coder [kind="agent"
    llm_model="claude-sonnet-4-6"
    prompt="Implement the plan.\nUse best practices."];
  Review [kind="agent"
    auto_status="true"
    prompt="Review the code.\nSet STATUS: success or fail."];

  Planner -> Coder;
  Coder -> Review;
  Review -> Done [condition="context.outcome==success"];
  Review -> Coder [condition="context.outcome==fail"
    restart="true"];
}

Features
Everything a pipeline needs

First-class syntax for the things that matter. Not string attributes on a graph node.

¶

Multi-line prompts

Indented blocks with zero escaping. Write real prompts, preserve blank lines, embed variables like ${ctx.input}.

◊

Typed node kinds

agent, tool, human, parallel, fan_in, subgraph. Not shape overloading.

✓

31 diagnostics

Structural validation and semantic lint. Catches dead edges, unreachable nodes, missing prompts, invalid models. Things DOT silently ignores.

⇝

Parallel execution

Native parallel fan-out and fan_in join with per-branch model overrides. Multi-provider consensus patterns in a few lines.

⌖

Scenario testing

dippin test injects context, simulates paths, and checks assertions. CI-ready JSON output. Test every conditional branch.

≡

Full toolchain

Format, lint, export-dot, migrate from DOT, cost estimation, coverage analysis, optimization, health reports, LSP server.

In action
