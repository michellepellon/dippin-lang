Scenario Testing

Test your AI pipeline workflows without deploying. Inject context values, simulate execution paths, and assert outcomes — all from a JSON file.

How It Works

The test runner loads scenarios from a .test.json file, injects context into the simulator, walks the execution graph, and checks assertions against the result.

.test.json
→
Load & Parse
→
Inject Scenario
→
Simulate Workflow
→
Check Assertions
→
PASS
/
FAIL

Test files are auto-discovered by convention: pipeline.dip looks for pipeline.test.json in the same directory.

Test File Format

Each test case declares a name, an optional scenario (context key-value pairs), and an expect block with assertions. 

{
  "tests": [
    {
      "name": "descriptive test name",
      "scenario": {
        "key": "value"
      },
      "expect": {
        "status": "success",
        "visited": ["NodeA", "NodeB"],
        "not_visited": ["NodeC"],
        "path_contains": ["NodeA", "NodeB"],
        "immediately_after": {"NodeA": "NodeB"}
      }
    }
  ]
}

Expectation Fields

All expectation fields are optional. Only specified fields are checked.

FieldTypeDescription
status string Expected simulation status: "success" (reached exit), "fail", or "dead_end"
visited string[] Node IDs that must appear in the execution path
not_visited string[] Node IDs that must NOT appear in the execution path
path_contains string[] Node IDs that must appear in order (non-contiguous matches allowed)
immediately_after object Map of {"A": "B"} pairs asserting B is the immediate next node after A

Full Example

A conditional routing workflow and its test file side by side.

gate.dip 
workflow Gate
  goal: "Route based on outcome"
  start: Start
  exit: Exit

  agent Start
    label: Start

  agent Pass
    model: claude-sonnet-4-6
    prompt: Handle success.

  agent Fix
    model: claude-sonnet-4-6
    prompt: Handle failure.

  agent Exit
    label: Exit

  edges
    Start -> Pass  when ctx.outcome = success
    Start -> Fix   when ctx.outcome = fail
    Pass -> Exit
    Fix -> Exit
gate.test.json 
{
  "tests": [
    {
      "name": "success path",
      "scenario": {
        "outcome": "success"
      },
      "expect": {
        "status": "success",
        "visited": ["Start", "Pass", "Exit"],
        "not_visited": ["Fix"]
      }
    },
    {
      "name": "failure path",
      "scenario": {
        "outcome": "fail"
      },
      "expect": {
        "status": "success",
        "visited": ["Start", "Fix", "Exit"],
        "not_visited": ["Pass"],
        "immediately_after": {
          "Start": "Fix"
        }
      }
    }
  ]
}

Caveats

Loop breaking and not_visited

The test runner limits per-node visits to 3. When a loop exceeds this limit, the simulator forces the loop-exit edge and continues execution. Nodes downstream of the loop exit can still be visited. For workflows with loops, prefer path_contains (checks ordering) over not_visited (can be fragile when the simulator continues past a forced loop exit).

Prefer immediately_after for edge routing

When testing which specific edge a node takes, immediately_after is more precise than path_contains. Use it to verify that a conditional edge routes to the expected next node.

Tool node defaults

Tool nodes auto-seed ctx.tool_stdout and ctx.outcome to "success". To test unconditional fallback edges after a tool node, set the key to an empty string in your scenario to suppress the default.
