Key Design Decisions

Everything flows through ir.Workflow

The ir package defines the canonical data model. Every tool — validator, formatter, exporter, simulator — programs against these structs. The IR contains no syntax details and no DOT shapes. It is a pure semantic representation.

Packages import ir, not each other

Most packages depend only on ir. The analysis packages (doctor, optimize, diff) compose other analysis packages, but core packages like parser, formatter, and export are fully independent. Zero external dependencies for core packages.

Condition.Parsed is populated lazily

The parser stores edge conditions as Condition.Raw (plain text). Condition.Parsed (AST) is only populated by simulate.EnsureConditionsParsed(). Any code reading parsed conditions must call this first — Lint() does it automatically.

Parallel branch simulation

The simulator handles parallel fan-out by visiting all branch nodes before proceeding to the fan_in join point. Path enumeration explores every branch combination for --all-paths mode. Events are emitted as JSONL for machine consumption.

The IR Package

The ir package is the heart of the system. It defines the data model that every other package operates on.

Design Principles

Explicit over implicit — Node.Kind directly states what the node is. Workflow.Start and Workflow.Exit are named fields, not inferred from declaration order.

Type-safe — Each node kind has its own config struct implementing a sealed NodeConfig interface. Invalid field combinations are structurally impossible.

Syntax-independent — The IR contains no Dippin syntax details and no DOT shapes. It is a pure semantic representation that could be produced by any frontend.

Sealed Interfaces

Both NodeConfig and ConditionExpr use Go's sealed interface pattern — an unexported method prevents external packages from creating invalid implementations: 

type NodeConfig interface { nodeConfig() }   // unexported method
type AgentConfig struct { ... }
func (AgentConfig) nodeConfig() {}            // only ir-package types can implement

The CLI

The CLI (cmd/dippin/cli.go) is a thin orchestration layer: parse global flags, dispatch to a command handler, load the workflow, call the appropriate package function, and render output.

Testability — The Run function accepts args []string and io.Writer, making it fully testable without touching os.Args or os.Stdout.

Auto-detection — loadWorkflow checks the file extension: .dot routes to migrate.Migrate, .dip routes to the Dippin parser.
