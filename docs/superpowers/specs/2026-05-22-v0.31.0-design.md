# v0.31.0 — `yes_no` human mode + tool-as-start round-trip (issues #45, #49)

**Date:** 2026-05-22
**Closes:** #45, #49
**Release target:** v0.31.0
**Predecessor:** v0.29.0 (#42 #43 #44) added the tool-routing follow-up surface; #49 was surfaced during that work and parked behind the v0.30.0 coverage-redirection fix (#40).

## Motivation

Two independent, small dippin-internal fixes:

1. **`mode: yes_no` rejected on human nodes (#45).** Tracker's runtime supports `yes_no` as a documented human mode, and the brief lists it. Dippin's validator does not — `validHumanModes` ships three entries and emits DIP127 for the fourth. Authors writing pipelines per the documented language surface hit a false-positive warning that drags doctor grades. Discovered while running the quality pass on `2389-research/pipelines` PR #13 — five `mode: yes_no` nodes in `interactive/story-engine.dip` all tripped DIP127.
2. **Tool node as workflow `start:` round-trips as agent (#49).** Surfaced during v0.29.0's #44 work. DOT export overrides the diamond shape to `Mdiamond` for the start marker; `migrate.resolveStartExitKind` recognizes manager_loop attrs on the recovered side but not tool attrs, so the migrated copy collapses `ToolConfig` to `AgentConfig` — `cfg.Outputs`, `MarkerGrep`, `Timeout` all disappear. The v0.29.0 round-trip test `TestRoundtripPreservesToolOutputs` uses an intermediate-agent workaround to dodge this exact bug.

Each is independently a patch; bundled because #49 touches IR-adjacent migration code and minor versioning is the cleaner story.

## Scope

Two independent code paths.

### #45 — `yes_no` is a valid human mode

- `validator/lint_human.go:28` — `Help` string becomes `"valid modes: choice, freeform, interview, yes_no"`.
- `validator/explanations.go:316-322` — DIP127 `Trigger` and `Fix` text picks up `yes_no`.
- `docs/validation.md:742` — sample diagnostic help string matches the new list.

Runtime semantics (whether `yes_no` is a distinct UX or an alias of `choice`) are tracker's call. Dippin's surface here is one entry in a static set.

### #49 — `hasToolConfigAttrs` recovers `NodeTool` after start-marker shape override

Mirror the existing `hasManagerLoopAttrs` pattern in `migrate/migrate.go`. Add a new heuristic:

```go
func hasToolConfigAttrs(attrs map[string]string) bool {
    toolKeys := []string{"tool_command", "marker_grep", "outputs", "route_required", "output_limit"}
    for _, key := range toolKeys {
        if _, ok := attrs[key]; ok {
            return true
        }
    }
    return false
}
```

Extend `resolveStartExitKind` (`migrate/migrate.go:325-330`) to consult it after the existing manager_loop check:

```go
func resolveStartExitKind(attrs map[string]string) ir.NodeKind {
    if hasManagerLoopAttrs(attrs) {
        return ir.NodeManagerLoop
    }
    if hasToolConfigAttrs(attrs) {
        return ir.NodeTool
    }
    return ir.NodeAgent
}
```

The five tool keys are unique to `ToolConfig` and cannot collide with any other node kind. `timeout` is intentionally excluded — it is shared with `HumanConfig` and would create false positives. The pathological "bare tool with only `command:`, exported without `IncludePrompts`" case is not recoverable under this heuristic; deferred (see Non-goals).

Complexity check: `hasToolConfigAttrs` is a single loop with a single conditional (cyclo 2). The expanded `resolveStartExitKind` is two ifs and a return (cyclo 3). Both well under the 5 / 7 envelope.

## Non-goals

- **Broader round-trip audit.** Human, subgraph, parallel, and fan-in node kinds may share the same start-marker round-trip gap. Deferred — separate issue if any actually break under exercise.
- **`yes_no` UX semantics.** Whether `yes_no` renders distinctly from `choice` in the tracker TUI is the runtime's call. Dippin's surface is the static valid-mode set.
- **Touching #41.** Parked. Terror-squad findings in `docs/superpowers/research/2026-05-19-issue-41-terror-squad.md` stand.
- **Explicit `kind=...` DOT attribute scheme.** Considered as Approach B during brainstorming; rejected for v0.31.0 because the heuristic mirrors the manager_loop precedent (smaller diff, no new DOT surface). If a future broader audit shows the bandaid pile growing, that's the right moment to flip the whole scheme to explicit kind attrs.

## Backwards compatibility

- `.dip` files that previously hit DIP127 on `mode: yes_no` will now lint clean. No file that previously passed becomes a failure.
- DOT round-trip: tool nodes used as `start:` previously silently collapsed to `AgentConfig`. They now preserve `NodeTool` and the full `ToolConfig`. No file that previously round-tripped cleanly is broken.
- No DOT output format change — the heuristic operates on existing attrs (`tool_command`, `marker_grep`, `outputs`, `route_required`, `output_limit`), all of which v0.29.0 already emits.

## Runtime requirement

None. Both fixes are dippin-internal; tracker is unaffected.
