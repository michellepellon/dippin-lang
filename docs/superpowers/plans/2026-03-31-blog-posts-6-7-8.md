**Content sections** (all build on a single running example):

**Section 1: The Linear Baseline** (use `<span class="step-num">1</span>` numbering) — A simple 3-node pipeline. Show the full `.dip` file:

```
workflow ReviewPipeline
  goal: "Draft, review, and publish a document"
  start: Start
  exit: Exit

  defaults
    provider: anthropic
    model: claude-sonnet-4-6

  agent Start
    label: Start

  agent Exit
    label: Exit

  agent Draft
    label: "Write Draft"
    prompt:
      Write a clear, concise technical document based on the
      provided requirements. Focus on accuracy and readability.

  agent Review
    label: "Review Draft"
    auto_status: true
    prompt:
      Review the draft for technical accuracy, clarity, and
      completeness. Return success if the draft meets standards,
      or fail with specific feedback.

  agent Publish
    label: Publish
    prompt:
      Format the approved draft for publication.

  edges
    Start -> Draft
    Draft -> Review
    Review -> Publish
    Publish -> Exit
```

One paragraph: "This works, but every draft gets published regardless of the review. We need branching."

**Section 2: Basic Conditions** (step-num 2) — Add `when` conditions to the Review edges:

```
  edges
    Start -> Draft
    Draft -> Review
    Review -> Publish  when ctx.outcome = success
    Review -> Draft    when ctx.outcome = fail
    Publish -> Exit
```

Show the lint output (clean — no DIP101 because success/fail is exhaustive). Explain `auto_status: true` — the LLM sets `ctx.outcome` to `success` or `fail`. Explain the retry loop: failed reviews go back to Draft.

**Section 3: Operators** (step-num 3) — Show each operator with a one-line edge example. Use a `<table>` for clarity:

| Operator | Example |
|----------|---------|
| `=` / `==` | `when ctx.outcome = success` |
| `!=` | `when ctx.outcome != success` |
| `contains` | `when ctx.feedback contains "security"` |
| `not contains` | `when ctx.feedback not contains "security"` |
| `startswith` | `when ctx.category startswith "urgent"` |
| `endswith` | `when ctx.filename endswith ".go"` |
| `in` | `when ctx.priority in high,critical` |

One paragraph: "All operators work on string values. The left side is always a context variable, the right side is a literal."

**Section 4: Compound Conditions** (step-num 4) — Show one example combining conditions:

```
Review -> Escalate  when ctx.outcome = fail and ctx.feedback contains "security"
Review -> Draft     when ctx.outcome = fail and ctx.feedback not contains "security"
```

Brief note about parentheses for precedence with `or`:
```
when (ctx.outcome = fail and ctx.severity = high) or ctx.override = true
```

**Section 5: Exhaustive Detection** (step-num 5) — This is the key insight. Show what happens with only one conditional edge:

```
  Review -> Publish  when ctx.outcome = success
  Review -> Exit
```

```
$ dippin lint pipeline.dip
PASS  pipeline.dip  (0 errors, 0 warnings)
```

Wait — no DIP101? Explain: the unconditional `Review -> Exit` acts as the fallback. Then show what triggers DIP101:

```
  Review -> Publish  when ctx.outcome = success
```

(No fallback, no complementary condition.)

```
$ dippin lint pipeline.dip
WARN  pipeline.dip
  DIP101: node "Review" has conditional edges but no unconditional fallback
```

Then show the fix — add the complementary condition:

```
  Review -> Publish  when ctx.outcome = success
  Review -> Draft    when ctx.outcome = fail
```

Clean lint. Explain: `success`/`fail` and `contains X`/`not contains X` are recognized as exhaustive pairs.

**Section 6: The ctx. Prefix** (step-num 6) — Show the DIP120 mistake:

```
  Review -> Publish  when outcome = success
```

```
$ dippin lint pipeline.dip
WARN  pipeline.dip
  DIP120: condition references "outcome" without namespace prefix — did you mean "ctx.outcome"?
```

One-character fix: add `ctx.`. Brief explanation of namespaces (ctx, graph, params).

```
═══ Cost Estimate ═════════════════════════════════════════
```

Walk through the output: three columns (min/expected/max), by-provider breakdown, top cost drivers sorted by max cost. Point out that `RemeasureCognitive` is $0.00 — that's a tool node (shell command, no LLM call).

**Section 2: How Estimation Works** (step-num 2) — Brief: "The cost estimator counts your prompt tokens (the text in your `prompt:` block maps to input tokens), applies a heuristic multiplier for expected output tokens, and multiplies by the model's per-token pricing. It's an estimate, not a bill — but it's in the right ballpark."

**Section 3: The max_turns Multiplier** (step-num 3) — Explain that nodes in retry loops get multiplied by `max_retries`. Show the `Refactor` node at $1.46 max — that's because it's in a retry loop. "If a node retries 3 times, the max cost reflects all 3 attempts. The expected cost assumes it succeeds on the first try."

**Section 4: Free Nodes** (step-num 4) — One paragraph: "Tool nodes and human nodes cost $0 — they don't make LLM calls. You'll see them in the cost breakdown with `$0.00` and no provider. Don't worry about them."

**Section 5: dippin optimize** (step-num 5) — Run optimize on the same workflow. Show the real output:

```
  Savings:   $0.63 (expected)

─── Suggestions ───────────────────────────────────────────
  [Refactor] node in retry loop — consider a cheaper model for mechanical iterations
    claude-opus-4-6 → claude-haiku-4-5  (saves ~$0.23)
  [Refactor] bookkeeping task (summary/cleanup/commit) can use a cheaper model
    claude-opus-4-6 → claude-haiku-4-5  (saves ~$0.23)
  [QualityGate] bookkeeping task (summary/cleanup/commit) can use a cheaper model

    claude-sonnet-4-6 → claude-haiku-4-5  (saves ~$0.04)
```

Walk through: current vs optimized total, then per-node suggestions. "The optimizer looks at each node's role and suggests where a cheaper model would do the same job. Retry loop iterations, bookkeeping tasks, and summary nodes are common targets."
